# 3장: 저수준 계층? 메모리라는 사물함에서부터 시작해보자

---

> [!NOTE]
>
> - 컴퓨터 시스템에서 연산과 저장을 담당하는 것은 CPU와 메모리
> - 메모리는 단순히 0과 1을 저장하는 사물함
> - 그런데 이 단순한 사물함에 **힙, 스택, 데이터, 가상 메모리, 메모리 할당, 메모리 해제, 메모리 누수** 같은 개념이 만들어짐
> - 어떤 프로그래밍 언어로 프로그램을 작성하든지 모든 프로그램은 메모리에서 실행되어야 하며, 여기에서 마주치는 문제들은 모두 매한가지다.

## 3.1 메모리의 본질, 포인터와 참조

**Byte (바이트)**

- **바이트는 8개의 비트를 묶은 메모리의 기본 단위다.**
- 8비트 = 2^8 = 256가지 조합
- 각 바이트마다 번호를 붙임 → **메모리 주소(memory address)**
- 예: `0x1000`, `0x1001`, `0x1002`...

```
메모리 (사물함)
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ 0 │ 1 │ 1 │ 0 │ 0 │ 1 │ 1 │ 1 │  (8 bits = 1 byte)
└───┴───┴───┴───┴───┴───┴───┴───┘
주소: 0x1000
```

**int (정수형)**

근데 256가지로는 부족하다!  
👉🏻 **4바이트를 묶어서 사용** = int (integer)

- 4 bytes = 32 bits = 2^32 = 약 42억 가지 조합
- 이게 `int` 형 변수의 크기가 4바이트인 이유!

```c
int age = 25;  // 4바이트 사용 (0x1000 ~ 0x1003)
```

**Structure (구조체)**

**구조체는 여러 개의 변수를 하나로 묶어 새로운 데이터 타입을 만드는 것이다.**

```c
struct Person {
    int age;      // 4 bytes
    float height; // 4 bytes
    char name[20]; // 20 bytes
};  // 총 28 bytes
```

- 더 복잡한 정보를 표현하려면 더 많은 바이트 필요
- 여러 바이트를 묶어서 의미 있는 데이터 구조 만듦

**Object (객체)**

**객체는 데이터(속성)와 동작(메서드)을 함께 묶은 프로그래밍 단위다.**

```javascript
const person = {
  age: 25, // 데이터
  name: "Alice", // 데이터
  greet() {
    // 동작 (메서드)
    console.log("Hi!");
  },
};
```

- 구조체 + 함수 = 객체
- 메모리 관점에서는 구조체와 비슷 (데이터 영역 + 함수 포인터)

### 3.1.2 메모리에서 변수로: variable

**우선 메모리 읽기와 쓰기의 본질에 대해서 알아보자**

**메모리 쓰기 (Write)**

`a = 1`

변수 `a`가 메모리 주소 `0x0006`에 할당되었다고 가정하면:

```
메모리
주소      값
0x0004   [  ]
0x0005   [  ]
0x0006   [ 1 ]  ← a가 여기 저장됨
0x0007   [  ]
```

"6번 주소에 값 1을 넣는다" = **메모리 쓰기**

**메모리 읽기 (Read)**

`b = a`

```
메모리
주소      값
0x0006   [ 1 ]  ← a 위치에서 값을 읽음
0x0008   [ 1 ]  ← 읽은 값을 b 위치에 씀
```

1. a의 주소(0x0006)로 가서 값 읽기 → `1`
2. 읽은 값을 b의 주소(0x0008)에 쓰기

👉🏻 **변수는 결국 "메모리 주소 + 값"일 뿐이다!**

- a는 값 1을 나타낸다
- 1은 주소 0x0006에 저장된다

### 3.1.3 변수에서 포인터로: pointer

> [!NOTE]
> 그런데 `"값"은 "메모리 주소"에 저장된다` 관점으로 살펴보면
>
> - 메모리 주소만 알고 있으면 해당 데이터를 찾을 수 있잖아?
> - 객체같이 메모리를 크게 차지하는 자료구조가 있다면?
>   - 굳이 불필요한 데이터의 복사본을 만들지 않고 주소를 저장하면 되지 않을까?

👉🏻 **포인터는 메모리 주소를 값으로 저장하는 변수다!**

```c
int a = 10;        // 일반 변수: 값(10)을 저장
int* ptr = &a;     // 포인터: 주소(0x0006)를 저장
```

**왜 포인터가 필요한가?**

1. **큰 데이터 복사 방지**
   - 객체가 1MB라면? 복사 대신 주소(8 bytes)만 전달
2. **같은 데이터 공유**
   - 여러 곳에서 같은 메모리를 가리킴
3. **동적 메모리 관리**
   - 힙 영역 메모리는 포인터로만 접근 가능

**간접 주소 지정 (Indirect Addressing)**

포인터의 본질 = 간접 주소 지정을 추상화한 것

```c
int b = 10;      // b는 0x3000에 저장
int* ptr = &b;   // ptr은 0x1000에 저장, 값은 0x3000
*ptr = 20;       // 간접 접근: ptr → b
```

```
어셈블리어 (저수준):
0x1000 → 0x3000 → 데이터(10)
모든 주소를 직접 관리

포인터 (고수준):
ptr → b
변수명만 쓰면 됨, 주소는 몰라도 됨
```

- **어셈블리어**: 변수 개념 없음 → 주소를 직접 다뤄야 함
- **포인터**: 간접 주소 지정을 변수로 감쌈 → "첫 번째 뜻(의도)"만 신경 쓰면 됨

👉🏻 **포인터는 복잡한 주소 계산을 숨기고 의도만 표현하게 해준다!**

### 3.1.4 포인터의 힘과 파괴력: 능력과 책임

- 주소라는 개념은 변수로 추상화 되었기 때문에 변수가 어디에 저장되는지 신경 쓸 필요 없이 이런 변수가 '존재한다' 라는 것만 알고 있으면 된다.
- 포인터가 있는 언어에서는 변수는 구체적으로 메모리 어느 위치에 저장되어있는지 직접 확인이 가능하다.

### 3.1.5 포인터에서 참조로: reference (메모리 주소 감추기)

- 포인터는 메모리 주소를 추상화한 것이고 참조는 포인터를 한 번 더 추상화 한 것이다.

**지하철 역으로 이해하기**

**포인터 = 역의 GPS 좌표 (37.497°N, 127.027°E)**

```
"강남역 가자"
→ GPS로 강남역 좌표 찾기: "37.497, 127.027"
→ 좌표가 진짜 있는지 확인 (null 체크)
→ 좌표로 이동
→ 만약 좌표가 잘못됐으면? (nullptr) → 길 잃음 💥
```

**참조 = 역 이름 (강남역)**

```
"강남역 가자"
→ 그냥 "강남역"이라고 하면 됨
→ 알아서 찾아감
→ 존재하지 않는 역 이름은 불가능 (항상 실제 역) ✅
```

**비교**

| 포인터 (GPS 좌표)             | 참조 (역 이름)            |
| ----------------------------- | ------------------------- |
| 좌표 확인 필요                | 이름만 말하면 됨          |
| 잘못된 좌표 가능 (위험)       | 존재하는 역만 가능 (안전) |
| 37.497, 127.027 → 찾기 (복잡) | 강남역 → 바로 (간단)      |

👉🏻 **참조는 "GPS 좌표(주소)"를 숨기고 "역 이름(변수명)"만 쓰게 만든 것!**

### Wrap Up: 추상화의 진화

**문제 1**: 메모리는 `0x1000` 같은 숫자 주소로만 접근 가능 (복잡함) → **해결**: 변수로 추상화해서 이름으로 접근 (`age`, `name`)  
**문제 2**: 큰 데이터를 복사하면 메모리 낭비 (1MB 객체를 계속 복사?) → **해결**: 포인터로 주소만 저장 (8 bytes로 해결)  
**문제 3**: 포인터는 `*`, `&`, null 체크 등 여전히 복잡함 (위험) → **해결**: 참조로 더 추상화해서 변수처럼 사용, null 불가능

👉🏻 **메모리 주소 → 변수 → 포인터 → 참조, 각 단계마다 복잡함을 숨기고 의도만 표현하게 만든 것!**

## 3.2 프로세스는 메모리 안에서 어떤 모습을 하고 있을까?

2주차에 프로세스는 실행 중인 프로그램이다. 라고 공부했음.
64비트 시스템에서 메모리 내 구조는 아래와 같다

```
High Address (0xFFFFFFFF)
┌──────────────────────────────┐
│       Kernel Space           │
├──────────────────────────────┤
│                              │
│          Stack               │  ← Function calls
│            ⬇️                │     Local variables
│                              │
├──────────────────────────────┤
│       (Free space)           │
├──────────────────────────────┤
│                              │
│            ⬆️                │
│           Heap               │  ← Dynamic memory
│                              │     (malloc, new)
├──────────────────────────────┤
│           Data               │  ← Global variables
│     (initialized data)       │     Static variables
├──────────────────────────────┤
│           Code               │  ← Program instructions
│          (Text)              │     (Read-only)
└──────────────────────────────┘
Low Address (0x00000000)
```

> [!NOTE]
>
> - 메모리의 모든 프로세스는 이런 모습을 하고 있음
> - 코드영역 / 데이터 영역 / 힙 영역 / 스택영역
> - 코드영역 / 데이터영역 -> 실행 파일을 초기화할 때 생성되는 영역
> - 힙 영역 -> 동적 메모리 할당에 사용
> - 스택영역 -> 함수 호출에 사용되며 매개변수, 반환 주소, 레지스터 정보 등을 포함한 함수 실행 시 정보를 저장하는데 사용

### 3.2.1~2 가상메모리: 눈에 보이는 것이 항상 실제와 같지는 않다.

- 프로세스가 사용하는 메모리 주소는 가상 주소다 (실제 물리 주소가 아님)
- 가상 메모리는 페이지 단위로 나뉘어 물리 메모리에 저장된다
- 페이지들은 물리 메모리 전체에 흩어져 있음 (연속되지 않음)
- 가상 주소 → 페이지 테이블 → 물리 주소 (변환 과정)

## 3.3 스택영역: 함수 호출은 어떻게 구현될까?

**JavaScript로 변환:**

```javascript
function func(a) {
  if (a > 1000000) {
    return;
  }

  const arr = new Array(100).fill(0); // 100개 배열
  func(a + 1); // 재귀 호출
}

func(0); // 💥 Stack Overflow!
```

**문제점: Stack Overflow (스택 오버플로우)**

- 함수 호출 시 Call Stack에 스택 프레임이 쌓임 (매개변수, 지역변수, 복귀 주소)
- `func(0)` → `func(1)` → `func(2)` → ... → `func(1000000)` 까지 **100만 번** 쌓임
- 각 프레임마다 배열 100개(400 bytes) + 기타 정보 = 수백 MB 메모리 사용
- 스택 영역 크기 한계(보통 1~8MB) 초과 → **Stack Overflow 에러**로 프로그램 종료
- 👉🏻 재귀 함수는 종료 조건 도달 전까지 계속 메모리를 쌓아서 위험!

**CPU가 함수를 호출하는 과정:**

1. **`call` 명령어 실행** → 현재 PC(다음 실행할 명령어 주소)를 스택에 푸시 = 반환 주소 저장
2. **새 스택 프레임 생성** → 매개변수, 지역변수, 반환 주소를 담는 공간 할당 (각 함수마다 독립적)
3. **매개변수 전달** → 스택 또는 레지스터에 값 복사 (호출 규약에 따라 다름)
4. **함수 실행** → 지역변수는 스택 프레임 내에 저장
5. **`return` 명령어** → 반환값을 레지스터(예: EAX)에 저장 → 스택 프레임 제거 → 저장된 반환 주소로 점프

## 3.4 힙영역: 메모리의 동적 할당은 어떻게 구현될까?

**왜 힙 영역이 필요한가?**

스택은 크기가 제한적이고(1~8MB) 함수 종료 시 자동으로 사라지며, 컴파일 타임에 크기를 알아야 함 → 큰 데이터나 런타임에 크기가 결정되는 데이터는 불가능

👉🏻 힙은 크기 제한이 크고(GB 단위), 프로그래머가 원할 때 할당/해제 가능, 함수 종료 후에도 유지됨 → 유연한 메모리 관리!

**힙 메모리 관리: Header-Payload-Footer + Doubly Linked List**

```
[Header|Payload|Footer] ⇄ [Header|Payload|Footer] ⇄ [Header|Payload|Footer]
 (블록1)                    (블록2-Free)              (블록3)
```

- **Header/Footer**: 블록 크기와 할당 여부 저장 → 양방향 탐색 가능 (Footer로 이전 블록 접근)
- **Doubly Linked List**: Free 블록들을 이중 연결 → `malloc()` 시 적절한 크기 빠르게 찾기
- **핵심 이점**: 메모리 해제 시 인접한 Free 블록을 **즉시 병합(coalescing)** 가능 → 메모리 단편화 방지
- 예: `[Used][Free1][Free2]` → Free1 해제 → `[Used][Free1+Free2]` (O(1) 병합)
- 👉🏻 결과: 빠른 할당/해제 + 메모리 효율성 향상!
