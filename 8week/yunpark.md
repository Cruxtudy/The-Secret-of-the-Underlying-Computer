## 6.1: CPU는 어떻게 입출력 작업을 처리할까?

- CPU 내부에 레지스터가 있는 것 처럼, 장치에도 자체 레지스터 (장치레지스터)가 있다!!
- 장치 레지스터는 장치 관련 정보를 저장하며 두 가지 종류가 있다
  - **데이터 레지스터**: 사용자가 키보드의 키를 누르면 그 정보가 저장됨
  - **상태/제어 레지스터**: 이 레지스터를 읽고 쓰는 작업으로 장치를 제어하거나 상태를 확인할 수 있음

> **Karabiner 예시** — 키보드 키 하나 누를 때
>
> 1. **데이터 레지스터** → 누른 키의 스캔코드 저장 (예: `caps_lock = 0x39`)
> 2. **상태/제어 레지스터** → "입력 있음" 표시 → OS가 읽어서 인터럽트 처리
> 3. **Karabiner** → OS가 꺼내온 그 값을 가로채서 `caps_lock`을 `ctrl`로 바꿔치기 한 뒤 내보냄
>
> 즉, Karabiner는 장치 레지스터를 직접 건드리는 게 아니라 OS가 읽어온 값을 **소프트웨어 레벨에서 remapping**하는 것

- 저수준 계층에는 두 가지 입출력 구현방법이 있다.

1. 특정 입출력 기계 명령어를 사용하는 것
2. 메모리의 읽기와 쓰기 명령어를 함께 사용하지만 주소 공간의 일부분을 장치에 할당하는 것

## 6.1.3: CPU가 키보드를 읽고 쓰는 것의 본질

```asm
Load R1, 0xFE00   ; 주소 0xFE00 (키보드 데이터 레지스터) 의 값을 R1 레지스터에 읽어옴
                  ; → 사용자가 누른 키의 스캔코드가 R1에 저장됨
```

- 키보드 데이터를 읽는 건 이게 전부다.
- 그런데 문제가 있다 — CPU는 사용자가 **언제** 키를 누를지 알 수 없다.

- 그래서 CPU는 장치의 작업 상태를 별도로 확인해야 하는데, 그 역할을 하는 게 바로 **장치 상태 레지스터**다.

---

## 6.1.4: 인터럽트 방식의 입출력 처리

CPU가 장치 상태를 확인하는 방법은 두 가지다.

**1. 폴링 (Polling)**
CPU가 상태 레지스터를 **주기적으로 직접 체크**함.
→ 키 입력이 없어도 계속 확인하므로 CPU 자원 낭비.

**2. 인터럽트 (Interrupt)** ← 실제로 쓰는 방식
장치가 할 일이 생기면 CPU에게 **먼저 신호를 보냄**.
CPU는 그때서야 하던 일을 멈추고 처리함.

> **키보드 입력 흐름 (인터럽트 방식)**
>
> 1. 사용자가 키를 누름 → 키보드 **데이터 레지스터**에 스캔코드 저장
> 2. 키보드가 CPU에 **인터럽트 신호** 전송
> 3. CPU는 현재 작업을 중단하고 **인터럽트 핸들러(ISR)** 실행
> 4. ISR이 `Load R1, 0xFE00`으로 데이터 레지스터 값을 읽어옴
> 5. 처리 완료 후 CPU는 하던 작업으로 복귀

핵심: CPU가 장치를 기다리는 게 아니라, **장치가 CPU를 부르는 구조**다.

## 6.2: 디스크가 입출력을 처리할 때 CPU가 하는 일은?

최신 컴퓨터 시스템에서 디스크 입출력은 CPU 개입 없이 처리된다.

디스크 입출력과 CPU 연산은 서로 독립적이기 때문에 병행 처리가 가능하다.

그렇다면 어떻게 CPU 개입 없이 동작하는가?
→ **장치 제어기**, **직접 메모리 접근(DMA)**, **인터럽트** 세 가지의 관계를 이해해야 한다.

---

## 6.2.1 장치 제어기

```
┌─────────────────────────────────────┐
│                 OS                  │
│         (소프트웨어 계층)               │
│                                     │
│  드라이버: 장치 제어기에 명령 전달          │
└──────────────┬──────────────────────┘
               │  명령 / 인터럽트 신호
               ▼
┌─────────────────────────────────────┐
│           장치 제어기                  │
│         (하드웨어 · 전자 부품)          │
│                                     │
│  • 데이터 레지스터                     │
│  • 상태/제어 레지스터                   │
│  • DMA: 메모리 ↔ 장치 데이터 이동        │
└──────────────┬──────────────────────┘
               │  전기 신호 / 기계 제어
               ▼
┌─────────────────────────────────────┐
│              장치                    │
│           (기계 부품)                 │
│                                     │
│  • 디스크: 모터, 헤드, 플래터            │
│  • 키보드: 스위치, 회로                 │
└─────────────────────────────────────┘
```

- **OS → 장치 제어기**: 드라이버가 제어 레지스터에 명령을 씀
- **장치 제어기 → 장치**: 전기 신호로 기계 동작 지시
- **장치 → 장치 제어기**: 완료 신호 / 데이터 전달
- **장치 제어기 → OS**: 인터럽트로 "작업 끝났어" 통보

## 6.2.2 CPU가 데이터를 복사해야할까?

- 아니다. **직접 메모리 접근(DMA)** 이 그 역할을 대신한다.
- DMA는 CPU를 거치지 않고 장치 ↔ 메모리 간 데이터를 직접 이동시킨다.
- CPU는 DMA가 끝났다는 인터럽트를 받고 나서야 개입한다.
