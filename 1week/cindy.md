# 1장: 프로그래밍 언어부터 프로그램 실행까지

---

## 1.1 여러분이 프로그래밍 언어를 발명한다면?

### 1.1.1 창세기: CPU는 똑똑한 바보

**CPU의 특징**
- CPU는 데이터를 한 곳에서 다른 곳으로 옮기고, 간단한 연산을 수행하는 작업만 할 수 있다
- 하지만 **엄청나게 빠른 속도**로 작업을 처리할 수 있다
- 초기 프로그래머들은 CPU의 관점에서 직접 **0과 1로 구성된 명령어(기계어)**를 작성해야 했다

### 1.1.2 어셈블리어의 등장

**어셈블리어란?**
- CPU가 실행할 수 있는 기계 명령어를 **인간이 읽고 이해할 수 있는 단어와 대응**시킨 언어
- 기계어와 특정 작업을 간단하게 대응시켜 만들어졌다
- 예시:
  ```asm
  sub $8, %rsp
  mov $.LC0, %edi
  call puts
  mov $0, %eax
  ```

**어셈블러의 역할**
- 어셈블리어를 0과 1로 구성된 바이너리 형태(기계어)로 변환하는 프로그램
- 처음으로 인간이 직접 인식할 수 있는 프로그래밍 언어가 탄생했다

### 1.1.3 저수준 계층의 세부 사항 대 고수준 계층의 추상화

**저수준 언어의 문제점**
- 어셈블리어는 여전히 저수준 언어로, **모든 세부 사항에 신경을 써야 한다**
- 비유: "앞으로 한 발짝 움직여"를 다음과 같이 표현해야 한다
  - "오른쪽 다리를 든다 → 오른쪽 무릎을 굽힌다 → 오른쪽 다리를 앞으로 내딛는다 → 오른쪽 무릎을 편다"

**해결 방안**
- 인간의 **추상적인 표현**을 CPU가 이해할 수 있는 **구체적인 구현**으로 자동 변환하는 방법이 필요하다
- 이를 통해 프로그래밍 생산성을 획기적으로 높일 수 있다

### 1.1.4 가득한 규칙: 고급 프로그래밍 언어의 시작

**공통 패턴 발견**

명령어들을 분석한 결과, 다음과 같은 공통 규칙들을 발견했다:

1. **문(statement)**: CPU에 특정 작업을 수행하라고 단도직입적으로 말하는 명령어
2. **조건문**: 조건에 따른 분기 (if-else)
3. **반복문**: 순환 구조 (while, for)
4. **함수**: 매개변수를 제외한 나머지 명령어를 묶어 코드로 지정

**예시**
```python
# 조건문
if 조건:
    실행문
else:
    실행문

# 반복문
while 조건:
    실행문

# 함수
def abc():
    실행문
```

**매개변수의 등장**
- 동일한 동작에 다른 매개변수를 넣어 다른 출력을 만들 수 있다
- 세부 사항과 함수를 분리할 수 있게 되었다

### 1.1.5 <인셉션>과 재귀: 코드의 본질

**재귀의 필요성**
- 문 안에 문이 들어가고, 단계 안에 단계가 포함되는 관계가 끝없이 이어진다
- 수열이 자식 수열에 의존하는 것처럼, 복잡한 구조를 몇 개의 간결한 문장으로 표현할 필요가 있다

**구문(syntax)**
- 재귀를 통해 끝없이 중첩된 것처럼 보이는 구조도 표현할 수 있다
- 세상의 모든 코드는 아무리 복잡하더라도 결국 **구문**으로 귀결된다
- 모든 코드가 구문에 기초하여 작성되기 때문이다

### 1.1.6 컴퓨터가 재귀를 이해하도록 만들기

**구문 트리(Syntax Tree)**
- 컴퓨터가 재귀 구문으로 표현된 문자열을 인식하려면 **트리 구조**로 표현해야 한다
- 재귀 구문에 따라 작성된 코드를 트리 형태로 구성할 수 있다
- 리프 노드(leaf node)의 표현이 매우 간단해져 기계 명령어로 번역하기 쉬워진다

**트리 구조의 장점**
- 복잡한 중첩 구조를 체계적으로 표현 가능
- 리프 노드부터 부모 노드로 올라가며 번역 가능

### 1.1.7 우수한 번역가: 컴파일러

**컴파일러의 정의**
- 프로그래머가 인간이 이해할 수 있는 언어로 작성한 코드를 **CPU가 이해할 수 있는 기계어로 변환**하는 프로그램
- 고수준 언어를 저수준 언어로 번역하는 역할을 담당한다

**컴파일러의 작동 방식**
1. 구문 정의에 따라 코드를 **트리 형태**로 구성
2. 리프 노드를 기계 명령어로 번역
3. 번역 결과를 차례대로 부모 노드에 적용
4. 최종적으로 전체 트리를 구체적인 기계 명령어로 변환

### 1.1.8 해석형 언어의 탄생

**플랫폼 독립성 문제**
- CPU에는 다양한 유형이 존재한다 (x86, ARM 등)
- 각 CPU는 자신만의 고유한 명령어 집합을 가지고 있다
- 플랫폼마다 다시 컴파일해야 하는 문제가 발생한다

**해결책: 가상 머신**
- **표준 명령어 집합**을 정의한다
- CPU의 기계 명령어 실행 과정을 **모방하는 프로그램**을 작성한다
- 각 CPU에 상응하는 시뮬레이션 프로그램을 준비하면 서로 다른 플랫폼에서 실행 가능하다

**인터프리터(Interpreter)**
- CPU 시뮬레이션 프로그램을 **가상 머신(Virtual Machine)** 또는 **인터프리터**라고 부른다
- 예: Java의 Bytecode를 실행하는 JVM (Java Virtual Machine)

**저수준 vs 고수준 언어**
- 저수준 언어: 성능·제어력 ↑
- 고수준 언어: 생산성·이해도 ↑
- 실무에서는 두 방식을 적절히 섞어 사용한다

## 1.2 컴파일러는 어떻게 작동하는 것일까?

### 1.2.1 컴파일러는 그저 일반적인 프로그램일 뿐

**소스 파일(Source File)**
- 프로그래머가 인간이 인식할 수 있는 단어로 코드를 작성한 일반 텍스트 파일
- 이 파일을 컴파일러에게 전달하면 실행 파일을 생성한다

**컴파일러의 역할**
- 소스 파일을 읽어 CPU가 직접 실행할 수 있는 기계 명령어가 작성된 실행 파일을 생성
- 일종의 번역기이자 텍스트 처리 프로그램이다

### 1.2.2 각각의 토큰 추출하기

**어휘 분석(Lexical Analysis)**
- 소스 코드를 탐색하며 모든 **토큰(Token)**을 찾아낸다
- 공백과 주석을 제거하고 토큰 단위로 쪼갠다

**토큰의 정의**
- 각 요소(키워드)와 추가 정보를 결합한 형태
- 예시:
  ```
  int a = 1;

  →

  T_Keyword    int
  T_Identifier a
  T_Assign     =
  T_Int        1
  ```

**스캐닝(Scanning)**
- 소스 코드에서 토큰을 추출하는 과정을 어휘 분석 또는 스캐닝이라고 한다

### 1.2.3 토큰이 표현하고자 하는 의미

**구문 분석(Syntax Analysis)**
- 토큰을 언어의 **문법 규칙**에 따라 구조적으로 분석한다
- 토큰들이 문법에 맞게 배치되었는지 확인한다

**해석(Parsing)**
- 컴파일러는 구문에 따라 토큰을 처리한다
- 예: `while` 키워드 뒤에는 `(` → bool 표현식 → `)` → `{` → 실행문 → `}` 순서로 와야 한다
- 문법에 맞지 않으면 문법 오류를 보고한다

**구문 트리 생성**
- 구문 규칙에 따라 토큰을 해석한 후 **구문 트리(Syntax Tree)**를 생성한다
- 이 전체 과정을 구문 분석이라고 부른다

### 1.2.4 생성된 구문 트리에 이상은 없을까?

**의미 분석(Semantic Analysis)**
- 구문 트리가 생성된 후 논리적 오류나 모순이 없는지 확인한다
- 구문은 맞지만 의미가 맞는지 검사한다
- 예: 타입 검사, 선언 확인, 정수형 변수와 문자열을 더하는 오류 등

### 1.2.5 구문 트리를 기반으로 중간 코드 생성하기

**중간 코드(IR Code, Intermediate Representation)**
- 의미 분석이 끝나면 구문 트리를 탐색하여 **좀 더 다듬어진 형태의 중간 코드**를 생성한다
- 기계어보다는 추상적이지만, 고급 언어보다는 하드웨어 친화적이다
- 최적화 작업을 수행하기 쉬운 형태이다

**중간 코드 예시**
```
a = 1
b = 2
goto B
A: b = b - 1
B: if a < b goto A
```

**중간 코드의 장점**
- 플랫폼 종속적이지 않음
- 최적화가 용이함
- 예: Java → Bytecode

### 1.2.6 코드 생성

**코드 생성 과정**
1. 중간 코드를 **어셈블리어(Assembly Code)**로 변환
2. 어셈블리어를 **기계 명령어(Machine Code)**로 변환
3. 기계 명령어는 **대상 파일(Object File)**에 저장된다

**컴파일 전체 흐름**
```
소스 파일
 → 어휘 분석 (Lexical Analysis)
 → 구문 분석 (Syntax Analysis / Parsing)
 → 의미 분석 (Semantic Analysis)
 → 중간 코드 생성 (IR Code)
 → 최적화
 → 코드 생성
 → 어셈블리어 변환
 → 기계어 변환
 → 대상 파일(Object File) 생성
```

**대상 파일의 특징**
- 각 소스 파일마다 하나의 대상 파일이 생성된다
- 확장자: `.o` (Linux), `.obj` (Windows)
- 여러 대상 파일을 하나의 실행 파일로 합치는 **링크 작업**이 필요하다

## 1.3 링커의 말할 수 없는 비밀

### 1.3.1 링커는 이렇게 일한다

**링커(Linker)의 역할**
- 컴파일러가 생성한 **여러 개의 대상 파일(Object File)**을 하나로 묶어 **최종 실행 파일(Executable File)**을 생성한다
- 여러 저자가 챕터별로 집필한 책을 한 권으로 묶어 출판하는 것과 비슷하다

**재배치(Relocation) 과정**
- 특정 소스 파일에서 다른 모듈에 정의된 함수를 참조할 때 문제가 발생한다
- 컴파일러는 해당 함수가 어느 메모리 주소에 위치할지 정확히 알 수 없다
- 컴파일러는 이를 표시(N)만 해두고 넘어간다
- 링커가 실행 파일을 생성하는 과정에서 함수의 정확한 주소를 확인하고 N을 실제 메모리 주소로 대체한다

### 1.3.2 심벌 해석: 수요와 공급

**심벌(Symbol)이란?**
- 전역 변수(Global Variable)와 함수(Function)의 이름을 포함하는 모든 변수 이름
- 지역 변수는 모듈 내에서만 사용되어 외부 모듈에서 참조할 수 없기 때문에 링커의 관심 대상이 아니다

**대상 파일의 구성**
1. **코드 영역(Code Section)**: 소스 파일에 정의된 함수에서 변환된 기계 명령어가 저장
2. **데이터 영역(Data Section)**: 소스 파일의 전역 변수가 저장
3. **심벌 테이블(Symbol Table)**: 정의/참조된 모든 심벌 정보

**심벌 테이블**
- 컴파일러는 외부 심벌 정보를 기록하는 심벌 테이블을 작성한다
- 두 가지 정보를 기록:
  1. **공급(Exported Symbols)**: 내가 정의한 심벌, 즉 다른 모듈에서 사용할 수 있는 심벌
  2. **수요(Imported Symbols)**: 내가 사용하는 외부 심벌

**심벌 해석(Symbol Resolution)**
- 링커는 각 대상 파일을 탐색하며 수요가 공급으로 충족되는지 확인한다
- 참조된 외부 심벌이 실제로 정의되어 있는지, 단 하나만 존재하는지 확인한다
- 수요가 공급을 초과하면 `undefined reference` 오류가 발생한다
- 공급이 수요를 초과하는 것은 허용된다

### 1.3.3 정적 라이브러리, 동적 라이브러리, 실행 파일

**정적 라이브러리(Static Library)**

정의:
- 코드를 별도로 컴파일한 후 패키지로 묶고, 구현된 모든 함수의 선언을 포함하는 헤더 파일을 제공하는 방식
- 파일 형식: `.lib` (Windows), `.a` (Linux)

정적 링크(Static Linking):
- 컴파일 단계에서 정적 라이브러리를 **실행 파일에 복사**하는 방식
- 소스 파일마다 단독으로 컴파일한다
- 대상 파일의 코드 영역끼리, 데이터 영역끼리 각각 결합한다

장점:
- 실행 파일이 독립적으로 동작 (외부 의존성 없음)
- 실행 속도가 빠름

단점:
- 여러 프로그램이 동일 라이브러리를 각각 복사 → **디스크와 메모리 낭비**
- 라이브러리 코드가 수정될 경우 **모든 프로그램을 다시 컴파일**해야 함

**동적 라이브러리(Dynamic Library)**

정의:
- 실행 파일에 라이브러리 전체가 복사되지 않고, **필수 정보만 포함**된다
- 공유 라이브러리(Shared Library)라고도 부른다
- 파일 형식: `.dll` (Windows), `.so` (Linux), `.dylib` (macOS)

동적 링크(Dynamic Linking):
- 실행 파일에는 다음 정보만 저장된다:
  - 참조된 동적 라이브러리 이름
  - 심벌 테이블
  - 재배치 정보
- 실제 프로그램이 **실행될 시점**에 라이브러리를 로드한다

**동적 링크의 두 가지 방식**

**(1) 프로그램이 메모리에 적재될 때 동적 링크 진행**
1. 실행 파일을 디스크에서 읽어 메모리의 특정 영역으로 이동 (적재, Loading)
2. 적재 도구가 실행 파일이 동적 라이브러리에 의존하는지 확인
3. 동적 링커가 실행되어:
   - 참조하는 동적 라이브러리 존재 여부와 위치 확인
   - 심벌의 메모리 위치 확인
   - 링크 과정 완료
4. 프로그램 시작

- 어떤 동적 라이브러리를 참조하는지 컴파일러에 명시적으로 알려줘야 한다

**(2) 런타임 동적 링크(Runtime Dynamic Linking)**
- 프로그램이 먼저 실행된 후, 실행 시간(runtime) 동안 코드가 직접 동적 링크를 실행
- 실행 파일 내부에 동적 라이브러리 정보가 저장되지 않는다
- 프로그래머가 코드에서 특정 API(`dlopen`, `dlsym`, `dlclose` 등)를 사용하여 필요할 때마다 동적 라이브러리를 직접 적재
- 실행 시간: CPU가 프로그램을 실행하기 시작한 시점 ~ 종료된 시점

### 1.3.4 동적 라이브러리의 장단점

**장점**

1. **리소스 절약**
   - 동적 라이브러리의 복사본은 디스크에 하나만 존재
   - 메모리에 적재되는 동적 라이브러리의 코드도 모든 프로세스가 하나의 코드를 공유
   - 메모리와 디스크 저장 공간을 대폭 절약

2. **유지보수 용이**
   - 동적 라이브러리 파일의 코드가 수정될 때 해당 동적 라이브러리만 다시 컴파일하면 됨
   - 프로그램 업그레이드와 버그 수정이 매우 간편

3. **플러그인 시스템 구현**
   - 플러그인에 필요한 함수들을 구현해두고, 프로그램 실행 시간에 플러그인을 적재하여 사용
   - 확장성 높은 시스템 구현 가능

4. **다언어 혼합 개발**
   - 높은 성능이 요구되는 부분은 C로 작성하여 컴파일 후 동적 라이브러리 생성
   - 나머지 부분은 Python 등으로 작성
   - Python에서 C언어 동적 라이브러리 함수를 호출하여 언어를 혼합한 개발 가능

**단점**

1. **성능 저하**
   - 프로그램이 적재되는 시간 또는 실행 시간에 링크되기 때문에 정적 링크보다 성능이 약간 떨어짐

2. **절대 주소 참조 불가**
   - 동적 라이브러리는 메모리에 단 하나의 복사본만 존재하고 여러 프로세스가 공유
   - 프로세스마다 다른 주소 공간을 가지므로 임의의 메모리 절대 주소로 참조할 수 없음
   - 간접 접근 방식 필요

3. **동적 라이브러리 종속성 문제**
   - 종속된 동적 라이브러리를 찾지 못하거나 버전이 호환되지 않을 경우 프로그램 실행 불가
   - DLL Hell 문제

### 1.3.5 재배치: 심벌의 실행 시 주소 결정하기

**재배치의 필요성**
- 컴파일러는 컴파일을 통해 대상 파일을 생성할 때 변수나 함수의 메모리 주소를 알 수 없다
- 프로그램이 실행되기 전까지 프로세스의 메모리 레이아웃이 확정되지 않기 때문이다

**재배치 테이블**

컴파일 단계:
1. 컴파일러는 메모리 주소를 알 수 없는 부분을 발견하면 간단하게 `0x00`으로 지정
2. 메모리 주소를 확정할 수 없는 변수를 발견할 때마다 재배치 테이블에 기록:
   - `.rela.text` (또는 `.relo.text`): 코드 영역의 명령어와 시작 지점으로부터의 주소
   - `.rela.data` (또는 `.relo.data`): 명령어 관련 데이터

링크 단계:
1. 링커가 심벌 해석 단계에서 대상 파일의 동일한 유형 영역끼리 병합 완료
2. 모든 기계 명령어와 전역 변수가 프로그램 실행 시간에 위치할 메모리 주소를 결정
3. 링커는 `.rela.text` 영역을 하나씩 읽어 심벌과 위치를 확인
4. 명령어를 찾고 이동할 소스 주소를 수정

**재배치(Relocation)**
- 링커가 실행 파일을 생성할 때 프로그램이 실행되는 시점에 함수가 적재될 메모리 주소를 확정
- 심벌의 메모리 주소를 수정하는 과정을 재배치라고 한다

### 1.3.6 가상 메모리와 프로그램 메모리 구조

**가상 메모리(Virtual Memory)의 역할**

문제:
- 변수나 명령어의 메모리 주소는 프로그램이 실행될 때마다 변경될 수 있다
- 실제로 그 시점이 되어야만 알 수 있다고 생각할 수 있다

해결:
- 가상 메모리는 각 프로그램이 자신만의 독립적인 메모리 공간을 사용하는 것처럼 착각하게 만든다
- 각 프로그램이 **모든 메모리를 독점적으로 사용**하는 것처럼 보인다
- 모든 프로그램은 동일한 표준적인 메모리 구조를 가질 수 있다

**메모리 주소 예측 가능성**
- 링커는 프로그램 실행 전에도 예측 가능한 메모리 구조를 기반으로 심벌 주소를 결정할 수 있다
- 프로그램 실행과 관계없이 메모리 구조를 예측할 수 있다
- 실행 시점에 심벌의 메모리 주소를 결정할 수 있다

**페이지 테이블(Page Table)**
- 가상 메모리는 물리 메모리와 1:1로 매핑되지 않는다
- 가상 메모리와 물리 메모리의 사상(mapping) 관계를 기록한 표를 페이지 테이블이라고 한다
- 각 프로세스는 자신만의 페이지 테이블을 가진다
- 하드웨어는 페이지 테이블을 참조하여 가상 주소를 물리 메모리 주소로 변환한다

**프로그램 메모리 구조**
- 모든 프로그램의 가상 메모리 주소는 `0x400000`에서 시작한다
- 일반적인 프로세스 주소 공간 구조:
  - 커널 영역
  - 스택(Stack) 영역
  - 힙(Heap) 영역
  - 데이터(Data) 영역
  - 코드(Code) 영역

## 1.4 컴퓨터 과학에서 추상화가 필요한 이유

### 추상화의 개념

**추상화(Abstraction)란?**
- 세부 사항을 모두 나열하지 않고도 의사소통을 할 수 있게 하는 개념
- 표현력을 크게 향상시키고 의사소통의 효율을 높인다
- 세부 사항을 노출할 필요가 없으므로 정보를 보호할 수도 있다

### 1.4.1 프로그래밍과 추상화

**소프트웨어의 복잡성 제어**
- 소프트웨어는 매우 복잡하지만, 추상화는 프로그래머가 이 복잡성을 효과적으로 제어할 수 있도록 돕는다
- 프로그래머는 추상화를 통해 복잡도를 제어할 수 있다

**프로그래밍 언어의 추상화 지원**
- 모든 프로그래밍 언어는 추상화를 지원하기 위해 각자 고유한 작동 방식을 제공한다
- 예: 객체 지향 언어는 다형성과 추상 클래스 등을 이용하여 추상화를 구현한다

### 1.4.2 시스템 설계와 추상화

**다양한 추상화의 예**

컴퓨터 시스템의 여러 요소가 추상화를 통해 단순화된다:

1. **CPU**: 명령어 집합(Instruction Set)으로 추상화
   - 하드웨어는 트랜지스터 여러 개로 구성
   - 명령어 집합이라는 개념으로 내부 구현 세부 사항 보호
   - 프로그래머는 명령어 집합에 포함된 기계 명령어만 사용하면 됨

2. **입출력 장치** → 파일(File)로 추상화
   - 파일 저장 내용과 위치 등 세부 사항을 신경 쓸 필요 없음

3. **실행 중인 프로그램** → 프로세스(Process)로 추상화

4. **물리 메모리와 파일** → 가상 메모리(Virtual Memory)로 추상화

5. **네트워크 프로그래밍** → 소켓(Socket)으로 추상화

6. **프로세스와 실행 환경** → 컨테이너(Container)로 추상화

7. **CPU, 운영체제, 응용 프로그램** → 가상 머신(Virtual Machine)으로 추상화

**추상화의 역할**
- 추상화는 프로그래머를 저수준 계층에서 점점 더 멀어지게 만든다
- 저수준 계층의 세부 사항에 신경 쓸 필요가 없도록 만든다
- 개발자는 더 높은 수준의 논리와 기능에 집중할 수 있다
- 소프트웨어 개발의 효율성과 생산성을 높일 수 있다

**추상화의 한계**
- 추상화 계층을 넘어가거나 자신만의 환경을 만들고 싶다면 필연적으로 저수준 계층에 대한 이해가 필요하다
- 프로그래밍의 질적 효율성을 높이려면 적절한 수준의 저수준 지식이 필요하다
