CPU, 컴파일러, 링커 기초 정리
1. CPU와 프로그래밍 언어의 탄생
1.1 CPU는 “똑똑한 바보”

CPU는 스스로 사고하지 못한다.
오직 0과 1로 구성된 기계 명령어(Instruction) 만을 순차적으로 실행하는 장치이다.

초기 컴퓨터는 프로그래머가 직접 기계어(0/1) 로 프로그램을 작성했다.

1.2 어셈블리어의 등장

기계어를 사람이 이해할 수 있도록 명령어에 이름을 붙인 언어가 어셈블리어다.

기계어	어셈블리
0101	ADD
1100	JMP

CPU 구조에 1:1 대응

인간이 처음으로 읽을 수 있는 프로그래밍 언어

1.3 저수준 vs 고수준

저수준 언어: CPU에 매우 가까움 (Assembly)

고수준 언어: 인간의 사고에 가까움 (C, Java, Python 등)

인간의 추상적 표현을 CPU가 이해할 수 있는 구체적 동작으로 변환하는 기술이 필요해짐.

1.4 고급 언어의 시작

고급 언어는 다음 개념을 도입했다.

함수(Function)

조건문(IF / ELSE)

반복문(WHILE / FOR)

매개변수(Parameter)

문제:

사람이 이해하는 문자열을 어떻게 CPU가 이해하는 기계 명령어로 바꿀 것인가?

1.5 재귀와 코드의 본질

모든 프로그램은 문법(Syntax) 에 따라 작성된다.
재귀 구조는 프로그램을 트리(Tree) 형태로 표현 가능하게 만든다.

1.6 컴파일러의 등장

컴파일러는:

고급 언어 → 기계어로 번역하는 프로그램


비유:

인간  ← 통역사(컴파일러) → 외계인(CPU)

1.7 인터프리터와 플랫폼 차이

x86 실행 파일은 ARM CPU에서 실행할 수 없다.

이유:

실행 파일 내부에 CPU 전용 명령어 가 들어있기 때문

빌드 대상	포함 명령어
x86	x86 Instruction
ARM	ARM Instruction
2. 컴파일러 동작 구조
2.1 컴파일러는 프로그램일 뿐이다

컴파일러는 소스 코드를 입력 받아 실행 파일을 출력하는 일반 프로그램이다.

2.2 어휘 분석 (Lexical Analysis)

소스 코드 → 토큰(Token) 분리

예:

int a = 3;
→ [int] [a] [=] [3] [;]

2.3 구문 분석 (Syntax Analysis)

토큰을 구문 트리(AST) 로 구성한다.

2.4 의미 분석 (Semantic Analysis)

타입 검사

선언 여부 확인

문법적/논리적 오류 검출

2.5 중간 코드 & 최적화

컴파일러는 중간 표현(IR) 을 만든 뒤 최적화를 수행한다.

예:

for(i=0;i<100;i++)
  x = 10 * 20;


→ 200 은 루프 밖에서 1회 계산 가능

효과:

연산 감소

캐시 효율 증가

CPU 파이프라인 최적화

2.6 코드 생성
AST → IR → Assembly → Machine Code

3. 링커(Linker)의 역할
3.1 링커란?

여러 개의 목적 파일(.obj, .o)을 하나의 실행 파일로 병합하는 도구.

3.2 심벌(Symbol) 해석

심벌 = 전역 함수 + 전역 변수

링커의 임무:

외부 심벌의 유일한 정의 탐색

주소 재배치(Relocation)

3.3 컴파일러 vs 링커
역할	컴파일러	링커
코드 생성	O	X
문법 검사	O	X
심벌 연결	X	O
실행 파일 생성	X	O
3.4 정적 라이브러리 vs 동적 라이브러리
구분	정적(.lib/.a)	동적(.dll/.so)
포함 시점	컴파일	실행
실행 파일 크기	큼	작음
메모리 공유	불가	가능
업데이트	재빌드 필요	즉시 반영
3.5 동적 링크 과정

Loader가 실행 파일 적재

Dynamic Linker 실행

DLL 탐색 및 심벌 연결

실패 시 실행 중단

4. 결론

CPU는 단순한 기계

컴파일러는 언어 번역기

링커는 모듈 결합기

고급 언어 = 생산성

저수준 제어 = 성능/OS 핵심
