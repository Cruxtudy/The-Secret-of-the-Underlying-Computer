### 어셈블리어 등장
- 어느날 프로그래머가 천공카드 뚫다가 아쉬움을 느껴버림.. CPU를 곰곰히 탐색해보는데
- 엥? 이거 "가산 명령어", "점프 명령어" 등 겨우 몇 가지 명령어만 실행할 수 있네?
- 이거 명령어들 싹 다 집합. 1:1 대응 시켜

```asm
sub  $8, %rsp      # 스택에 8바이트 공간 확보 (스택 포인터 감소)
mov  $.LC0, %edi   # .LC0 주소(문자열)를 edi 레지스터로 복사 → puts 함수의 첫 번째 인자
call puts          # puts 함수 호출 → 화면에 문자열 출력
mov  $0, %eax      # 0을 eax에 저장 → 함수 반환값 (return 0)
```

- CPU는 add, mov, call 같은 단순 명령어만 실행 가능
- 어셈블리는 이 명령어들을 사람이 읽을 수 있는 니모닉(mnemonic)으로 1:1 대응
- 기계어 10110000 01100001 → 어셈블리 mov $0x61, %al
- 그리고 이 기계 명령어를 0과 1로 변환하는 프로그램을 사용해서 CPU 저속 노화 식단으로 변환

> [!NOTE]
> 하지만 너무 아쉽다. 여전히 저수준 언어다.

  - 그럼 그 세부 사항이 뭔데?
  - 저수준 언어를 이용하여 프로그래밍을 하면 '데이터를 한곳에서 다른 곳으로 이동시키고, 간단히 연산한 후 또다시 다른 곳으로 이동시키는 것'처럼 몇 가지 간단한 명령어를 사용하여 정렬 같은 복잡한 문제를 구현해야 한다.

### 가득한 규칙: 고급 프로그래밍 언어의 시작
- 또 패턴을 찾아버렸다.
- 명령어는 대부분 단도직입적이다.
  - 좋다. 이제부터 이 **단도직입적인 명령어에 문(statement)라는 이름**을 붙히자
- 특정 상황에 따라 실행할지 결정해야 하는 선택이 필요하다
  - 좋다. 얘는 **'만약 ~라면 ~하고, 그렇지 않다면 ~한다"** 로 하자.
- **반복 상황**이 있네?
- 어? 근데 반복 상황에서 parameter만 바꿔서 이걸 별도로 분리하고 매개변수를 제외한 나머지 명령어를 하나로 묶을 수도 있겠는데?
  - -> 함수

- 재귀 -> 트리 구조로 표현 가능
  - 어? 그럼 리프 노드부터 시작하면 되겠네?
  - 리프 노드(맨 밑바닥)를 기계 명령어로 번역 → 그 결과를 부모 노드에 적용
  - 이걸 계속 위로 올라가면? 전체 트리가 기계 명령어로 쫙 변환됨
  - **bottom-up 방식으로 고수준 언어 → 기계어 자동 변환 완성**

### 해석형 언어 탄생
- 근데 문제가 생김. CPU마다 기계어가 다 다름
  - Intel CPU용으로 짠 코드 → AMD CPU에서 안 돌아감
  - 하드웨어 제조사가 만든거라 바꿀 수도 없음
  
- 어? 그럼 우리가 **서울말 CPU를 소프트웨어로 만들면** 되지 않나?
  - 표준 명령어 집합을 정의하고
  - 실제 CPU의 동작을 흉내내는 프로그램 작성
  - 이 프로그램이 코드를 한 줄씩 읽어서 실행
  
- 이게 바로 **가상머신(Virtual Machine)**
  - 별명: **인터프리터(Interpreter)**
  - 한 번 짜면 어떤 CPU에서든 돌아감 (이식성 획득)
