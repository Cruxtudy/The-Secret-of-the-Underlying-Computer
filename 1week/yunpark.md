### 어셈블리어 등장
- 어느날 프로그래머가 천공카드 뚫다가 아쉬움을 느껴버림.. CPU를 곰곰히 탐색해보는데
- 엥? 이거 "가산 명령어", "점프 명령어" 등 겨우 몇 가지 명령어만 실행할 수 있네?
- 이거 명령어들 싹 다 집합. 1:1 대응 시켜

```asm
sub  $8, %rsp      # 스택에 8바이트 공간 확보 (스택 포인터 감소)
mov  $.LC0, %edi   # .LC0 주소(문자열)를 edi 레지스터로 복사 → puts 함수의 첫 번째 인자
call puts          # puts 함수 호출 → 화면에 문자열 출력
mov  $0, %eax      # 0을 eax에 저장 → 함수 반환값 (return 0)
```

- CPU는 add, mov, call 같은 단순 명령어만 실행 가능
- 어셈블리는 이 명령어들을 사람이 읽을 수 있는 니모닉(mnemonic)으로 1:1 대응
- 기계어 10110000 01100001 → 어셈블리 mov $0x61, %al
- 그리고 이 기계 명령어를 0과 1로 변환하는 프로그램을 사용해서 CPU 저속 노화 식단으로 변환

> 하지만 너무 아쉽다. 여전히 저수준 언어다.

### 컴파일러는 대단하지 않다
```mermaid
flowchart LR
    A[소스코드] --> B[컴파일러] --> C[실행파일]
```

**1단계: 어휘 분석 (Lexical Analysis)**
- 소스코드를 쭉 읽으면서 토큰으로 쪼갬
- `int x = 10;` → `[int] [x] [=] [10] [;]`
- 단어를 하나하나 분리하는 단계

**2단계: 구문 분석 (Parsing)**
- 어? 토큰만 있으면 의미를 알 수 있나?
- ㄴㄴ. 토큰들의 관계를 파악해야 함
- 트리 구조(AST)로 표현 → 문법 구조 명확화

**3단계: 의미 분석 (Semantic Analysis)**
- 문법은 맞는데... 의미적으로 이상한 건 없나?
- 타입 체크, 변수 선언 확인 등
- 예: `int x = "문자열";` ← 문법은 맞지만 타입 에러

**4단계: 중간 코드 생성**
- AST를 기반으로 중간 표현 코드 생성
- 아직 특정 CPU용이 아님

**5단계: 코드 생성**
- 중간 코드 → 어셈블리어로 변환
- 이제 특정 CPU 아키텍처용 코드가 됨

> [!NOTE]
> 근데 과연 이걸로 진짜 끝났을까?

**ㄴㄴ 아직 아님**

컴파일 결과는 `code.o` 같은 **오브젝트 파일(object file)**로 저장됨  
각 소스 파일마다 `.o` 파일이 하나씩 생김

- 어? 그런데 우리가 원하는 건 **하나의 실행파일**인데?
- 맞음. 그래서 **링커(Linker)**가 등장
  - 여러 `.o` 파일들을 하나로 병합
  - 외부 라이브러리도 연결
  - 최종 실행 파일 생성


### 링커(Linker)가 뭔데?

소스 코드를 담고 있는 `func.c` 파일이 있다고 가정해보자. 이 파일을 컴파일하면 `func.o`라는 이름의 기계 명령어에 해당하는 코드를 저장하는 파일이 생성되는데, 이 파일을 대상 파일(object file)이라고 한다.
- 윈도우에서 흔히 보는 EXE 형식의 실행파일은 링크가 필요한 대상 파일을 한데 모아 구성한다.

> [!NOTE]
> 링크의 전체 과정은 저자 여러 명이 각각 특정 부분을 맡아 장(chapter)별로 따로 집필하고, 개별 장을 묶어 책 한 권으로 출판하는 것과 비슷하다!

- 근데, 이거 살펴보니까 링커랑 yarn.lock / pnp랑 공통점이 있는 것 같다.

```
C 프로그램:
  main.o + utils.o + libc.so
         ↓ [링커]
  실행 파일 (모든 심볼 해결됨)

Node.js 앱:
  express@4.18.0 + lodash@4.17.21 + react@18.0.0
         ↓ [yarn.lock/pnp]
  node_modules (모든 패키지 해결됨)
```

> [!NOTE]
> **재밌는 관찰**: 링커의 개념은 여러 레벨에서 반복된다
> - **컴파일 타임**: 링커 (`.o` → 실행파일)
> - **런타임**: 동적 링커 (`.so`, `.dll` 로드)
> - **패키지 레벨**: yarn.lock, pnp (패키지 의존성)
> - **모듈 레벨**: ES6 import, CommonJS require
> 
> 모두 "조각들을 찾아서 연결한다"는 같은 철학을 공유한다!

다시 돌아와서, 링커가 정확히 뭐임?

**1) 심볼 해석 (Symbol Resolution)**
- 책 비유: "3장 참고"라고 했는데, 3장이 실제로 있는지 확인
- 코드 버전: `printf()` 함수를 호출했는데, 실제 구현체가 어디 있는지 찾기
- 외부 심볼 참조 → 실제 구현 연결
- 만약 없으면? 링커 에러! `undefined reference to 'printf'`

**2) 재배치 (Relocation)**
- 책 비유: 집필 중엔 "N쪽 참고"라고 임시 표시 → 최종 편집 때 실제 페이지 번호로 교체
- 코드 버전: 함수 주소를 일단 `0x00000000` 같은 임시값으로 표시 → 링커가 실제 메모리 주소로 교체
- 왜 필요함? 컴파일할 땐 최종 메모리 주소를 모름

**예시:**
```c
// main.c에서
extern int add(int, int);  // "어디선가 있겠지~" 선언만
int result = add(1, 2);    // 주소? 모름. 일단 0x????

// utils.c에서
int add(int a, int b) { return a + b; }  // 실제 구현

// 링커가 하는 일:
// 1. add() 구현 찾기 (심볼 해석)
// 2. main.c의 0x???? → utils.c의 실제 주소로 변경 (재배치)
```

링커 = **"빠진 퍼즐 조각 찾아서 끼워넣기 + 주소 확정"**

### 심벌 해석: 수요와 공급

**오브젝트 파일(`.o`) 구조:**
```
소스파일 → 컴파일 → .o 파일
                      ├─ 코드 영역 (실제 명령어)
                      ├─ 데이터 영역 (변수들)
                      └─ 심벌 테이블 (import/export 정보)
```

**컴파일러의 전략:**
- 외부 함수/변수 발견? 신경 안 씀. 일단 패스
- "이거 어디 있을 거야~" 하고 표시만 해둠
- 실제로 찾는 건 링커 몫

**심벌 테이블 = 수요/공급 명세서**

링커 부담을 줄이기 위해 각 `.o` 파일에 메타데이터 기록:

1. **공급 (Export)**: 내가 제공하는 심벌
   ```c
   int add(int a, int b) { ... }  // ← 다른 파일에서 쓸 수 있음
   ```

2. **수요 (Import)**: 내가 필요한 심벌
   ```c
   extern int add(int, int);  // ← 어디선가 가져와야 함
   ```

> [!NOTE]
> 어? 이거 JavaScript의 `import`/`export`랑 똑같은 개념 아님?
> 
> ```javascript
> // utils.js
> export function add(a, b) { ... }  // 공급
> 
> // main.js
> import { add } from './utils.js';  // 수요
> ```
> 
> **맞음!** 링커의 심벌 테이블이 모든 모듈 시스템의 조상이다.  
> ES6 모듈, CommonJS, Python import... 전부 같은 철학.

**정리:**
- 심벌 테이블 = "내가 뭘 주고, 뭘 받는지" 명시
- 링커는 이걸 보고 퍼즐 맞추기
- 수요는 있는데 공급이 없으면? → `undefined reference` 에러

### 동적 링커 (Dynamic Linker) vs 정적 링커

**타이밍부터 다름:**

```
정적 링커:  컴파일 타임 (빌드할 때)
  main.o + utils.o + libc.a → 실행파일 (모든 코드 포함)
  
동적 링커:  런타임 (실행할 때)
  실행파일 실행 → "libc.so 필요하네?" → 메모리에 로드 → 연결
```

**정적 링킹 (Static Linking)**
- 모든 라이브러리 코드를 실행 파일에 **통째로 복사**
- 장점: 독립적. 다른 거 필요 없음
- 단점: 파일 크기 폭증

```bash
# 정적 링킹 예시
gcc main.c -static -o myapp
# → myapp 크기: 2MB (libc 전체 포함)
```

**동적 링킹 (Dynamic Linking)**
- 라이브러리는 **따로 보관**하고, 실행 시 연결
- 장점: 파일 작음, 메모리 절약
- 단점: `.so` 파일 없으면 실행 안 됨

```bash
# 동적 링킹 (기본값)
gcc main.c -o myapp
# → myapp 크기: 16KB (libc는 /lib/libc.so 참조만)

# 실행하면?
./myapp
# 1. OS가 동적 링커 호출 (Linux: ld.so, macOS: dyld)
# 2. 동적 링커가 libc.so 찾아서 메모리 로드
# 3. printf 같은 함수 주소 연결
# 4. 프로그램 실행
```

**왜 동적 링킹을 쓰냐?**

1. **메모리 절약**
   ```
   100개 프로그램이 printf 사용
   - 정적: libc 코드 × 100 = 메모리 낭비
   - 동적: libc.so 하나만 메모리에 로드 → 모두가 공유
   ```

2. **업데이트 편함**
   ```
   libc에서 보안 버그 발견!
   - 정적: 모든 프로그램 재컴파일 필요
   - 동적: libc.so만 업데이트 → 모든 프로그램에 자동 반영
   ```

3. **플러그인 시스템 가능**
   ```c
   // 런타임에 코드 로드
   void *handle = dlopen("plugin.so", RTLD_LAZY);
   void (*func)() = dlsym(handle, "plugin_init");
   func();  // 플러그인 실행!
   ```

**웹 개발로 비유하면?**

- **정적 링킹** = Webpack으로 번들링
  - 모든 코드를 `bundle.js`에 때려박음
  - 배포하면 끝

- **동적 링킹** = CDN에서 로드
  ```html
  <!-- 런타임에 외부에서 가져옴 -->
  <script src="https://cdn.com/react.js"></script>
  ```

ESM의 동적 `import()`도 동적 링킹과 같은 개념!

**정리:**

| | 정적 링커 | 동적 링커 |
|---|----------|----------|
| **시점** | 컴파일 타임 | 런타임 |
| **결과** | 실행파일에 모두 포함 | 실행 시 `.so` 로드 |
| **크기** | 큼 | 작음 |
| **의존성** | 독립적 | `.so` 파일 필요 |
| **업데이트** | 재컴파일 필요 | 라이브러리만 교체 |

### 가상 메모리: 링커가 주소를 미리 정할 수 있는 이유

**문제 상황:**
```
링커: "printf() 함수는 0x08048000 주소에 있다고 적어놨어!"
런타임: "어? 근데 실제 물리 메모리는 매번 달라지는데..."
```

링커가 컴파일 타임에 정한 주소 → 런타임의 실제 메모리 위치가 다르면 어떡하지?

**해답: 가상 메모리 (Virtual Memory)**

링커는 **실제 물리 주소를 모릅니다.** 대신 **가상 주소**를 씁니다.

**가상 주소 vs 물리 주소:**

```
[컴파일 타임 - 링커가 하는 일]
printf() → 가상 주소 0x08048000 할당
         (고정된 주소, 매번 동일)

[런타임 - OS가 하는 일]
가상 주소 0x08048000 
  → MMU(Memory Management Unit)가 변환
  → 실제 물리 주소 0x7F3A2000 (매번 다를 수 있음)
```

**왜 이게 필요한가?**

1. **예측 가능성**
   - 링커는 항상 같은 가상 주소로 작업 가능
   - 실제 물리 메모리 위치는 신경 안 써도 됨

2. **보안 (ASLR)**
   ```
   프로그램 A 실행: printf → 물리 주소 0x1234
   프로그램 A 재실행: printf → 물리 주소 0x5678 (다름!)
   
   하지만 가상 주소는 둘 다 0x08048000 (동일)
   ```

3. **메모리 격리**
   ```
   프로그램 A: 가상 주소 0x1000 → 물리 주소 0xAAAA
   프로그램 B: 가상 주소 0x1000 → 물리 주소 0xBBBB
   
   같은 가상 주소를 써도 충돌 안 남!
   ```

**동작 과정:**

```
1. [링커 타임]
   main.c: printf() 호출 → 가상 주소 0x???? 표시
   링커: "printf는 가상 주소 0x08048000이야" (재배치)

2. [로드 타임]
   OS: 실행파일을 메모리에 로드
   페이지 테이블 생성 (가상 → 물리 매핑)

3. [런타임]
   CPU: 가상 주소 0x08048000 접근 시도
   MMU: 페이지 테이블 조회 → 물리 주소 0x7F3A2000로 변환
   실제 메모리: 0x7F3A2000 위치의 코드 실행
```

**핵심:**
- 링커는 **가상 주소**를 사용 (고정됨, 예측 가능)
- OS/MMU가 **물리 주소**로 변환 (동적, 매번 다를 수 있음)
- 링커는 런타임의 실제 물리 주소를 **알 필요가 없음**

**웹 개발 비유:**
```javascript
// 링커 = DNS (도메인 → IP 매핑)
// 가상 주소 = 도메인 (example.com)
// 물리 주소 = IP 주소 (192.168.1.1)

// 코드에서는 항상 도메인 사용
fetch('https://example.com/api')

// 실제 IP는 DNS가 런타임에 결정
// CDN, 로드밸런서에 따라 매번 다를 수 있음
```

정리: **링커는 런타임 물리 주소를 모르고, 알 필요도 없습니다. 가상 메모리 덕분에!**

### 컴퓨터과학과 추상화

- 컴파일러와 링커는 컴퓨터 과학에서는 초석과도 같은 매우 중요한 역할을 맡고 있음
- 프로그래머가 기계 명령어와 같은 세부 사항에 신경 쓰지 않고 효율적으로 프로그래밍 할 수 있는 것은 추상화 때문!

**1. 입출력 장치 → 파일로 추상화**
```
USB, 하드디스크, SSD, 프린터... 다 다른데?
→ OS: "몰라, 다 파일이야"
→ 프로그래머: read(), write()만 쓰면 끝
```
- 마우스도 파일 (`/dev/mouse`)
- 키보드도 파일 (`/dev/keyboard`)
- 다 똑같이 다룸!

**2. 실행 중인 프로그램 → 프로세스로 추상화**
```
크롬 켜기, 카카오톡 켜기, 게임 켜기
→ OS: "얘네 다 프로세스야"
→ CPU 시간 나눠주고, 메모리 할당하고, 관리함
```
- 작업 관리자에 보이는 그거임
- PID(프로세스 ID)로 구분

**3. 물리 메모리 + 하드디스크 → 가상 메모리로 추상화**
```
램 8GB밖에 없는데 10GB 프로그램 실행?
→ OS: "하드디스크 좀 빌려서 가상으로 메모리 늘림"
→ 프로그래머: "메모리 무한대인 줄 알고 코딩"
```
- 실제론 램+하드 섞어서 씀
- 프로그래머는 그냥 주소만 씀

**4. 네트워크 → 소켓으로 추상화**
```
카톡 메시지 보내기, 롤 서버 연결, 유튜브 스트리밍...
→ OS: "다 소켓으로 통일"
→ send(), recv() 함수만 쓰면 됨
```
- WiFi든 LAN선이든 신경 안 써도 됨
- TCP, UDP 차이만 알면 끝

**5. 프로세스 + 실행 환경 → 컨테이너로 추상화**
```
"내 컴에선 되는데요?" ← 이거 없애기
→ Docker: "프로그램 + 설정 + 라이브러리 다 박스에 넣음"
→ 어디서든 똑같이 실행됨
```
- 예: Node.js 버전, 환경 변수 다 포함
- 배송 상자처럼 포장해서 이동

**6. CPU + OS + 프로그램 → 가상머신으로 추상화**
```
맥북인데 윈도우 프로그램 돌리고 싶다?
→ VMware: "가짜 컴퓨터를 소프트웨어로 만들어줌"
→ 맥북 위에서 윈도우가 "진짜 컴퓨터"인 줄 앎
```
- 컴퓨터 안에 또 컴퓨터
- AWS EC2도 이거 (클라우드 서버 = 가상머신)

**패턴 발견:**
> 복잡한 거 → 간단한 인터페이스로 포장
> 
> 프로그래머는 세부 사항 몰라도 됨 = 추상화의 힘!
